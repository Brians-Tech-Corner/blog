## The Problem with AI Code Review

Most AI code review tools are essentially very fast linters. They'll tell you about variable naming, flag obvious null pointer risks, and remind you to add error handling. What they won't do is ask: *"If the engineer who built this quit tomorrow, could someone else debug it in an incident?"*

That question — and a dozen like it — is what a Staff or Principal engineer brings to a review that a junior engineer, or an AI, typically doesn't. They've been paged at 3am. They've watched "we'll clean it up later" last five years. They know the difference between code that works and code that's safe to operate.

[Greybeard](https://github.com/btotharye/greybeard) is my attempt to bottle that perspective.

## What Is Greybeard?

Greybeard is a CLI tool and MCP server that reviews code, architecture decisions, and design docs through the lens of experienced engineers. You pipe a diff or document into it, and it responds with the questions your Staff reviewer would ask — not the ones a linter would.

```bash
# Review a git diff
git diff main | greybeard analyze

# Review with a specific perspective
cat design-doc.md | greybeard analyze --mode mentor --pack startup-pragmatist

# Self-check before sharing a decision
greybeard self-check --context "We're migrating auth to a new provider mid-sprint"
```

It works with whatever LLM you prefer — OpenAI, Anthropic Claude, or local models via Ollama and LM Studio. No mandatory cloud dependency.

## Content Packs: Swappable Perspectives

The key idea that makes Greybeard more than a wrapper around GPT is **content packs**. Each pack is a plain YAML file that defines a persona, tone, focus areas, heuristics, and example questions.

The built-in packs ship with it:

| Pack | Perspective | Example Question |
|---|---|---|
| `staff-core` | Staff Engineer — ops, ownership, long-term cost | "Who owns this if things break?" |
| `oncall-future-you` | On-call engineer at 3am — failure modes, pager noise | "What wakes us up at 3am?" |
| `mentor-mode` | Experienced mentor — teaching, not just reviewing | "What's the learning opportunity here?" |
| `solutions-architect` | SA lens — boundaries, fit-for-purpose | "Is this solving the right problem?" |
| `idp-readiness` | Platform Engineering — IDP maturity, automation | "Does this fit the golden path?" |
| `security-reviewer` | AppSec — auth flaws, injection, secrets, abuse surface | "What happens if a valid token from tenant A is used against tenant B?" |
| `startup-pragmatist` | Counter to over-engineering — is this complexity warranted? | "Will this matter in 6 months?" |
| `incident-postmortem` | SRE — blamelessness, root cause depth, action item quality | "What would we wish we'd done differently?" |
| `data-migrations` | Migration safety — locks, rollbacks, backfill blast radius | "Can we roll this back at 2am if it breaks?" |

The `security-reviewer` pack, for example, asks questions like:

> *"What happens if a valid token from tenant A is used against tenant B's resources?"*
> *"Does the error message reveal anything to an unauthenticated caller?"*
> *"Can an unauthenticated caller reach any path through this code?"*

These aren't linter rules — they're the questions an AppSec engineer asks from experience.

You can install community packs from GitHub:

```bash
greybeard pack install github:someone/their-greybeard-packs
```

Or write your own — a pack is just a YAML file:

```yaml
name: my-team-pack
description: Our team's specific review perspective
perspective: |
  An engineer who has operated our payment processing pipeline
  and knows exactly where the sharp edges are.
tone: precise and specific to our domain
focus_areas:
  - PCI compliance
  - idempotency of payment operations
  - audit log completeness
heuristics:
  - "Can this operation be replayed safely if it fails halfway through?"
  - "Is every state transition in the payment lifecycle logged?"
```

## MCP Server Integration

Greybeard also runs as an MCP server, which means it integrates with Claude Desktop, Cursor, Zed, and any other MCP-compatible tool:

```json
{
  "mcpServers": {
    "greybeard": {
      "command": "greybeard",
      "args": ["mcp"]
    }
  }
}
```

Once connected, your AI coding tool has access to `review_decision`, `self_check`, and `coach_communication` tools — bringing Greybeard's perspectives directly into your editor workflow.

## The Four Lenses

Whatever pack you use, Greybeard always reasons through four lenses:

1. **Operational impact** — failure modes, observability, deploy and rollback safety
2. **Long-term ownership** — who owns this in 6–12 months, tribal knowledge risk
3. **On-call and human cost** — pager noise, manual recovery, 3am failure scenarios
4. **"Who pays for this later?"** — complexity tax, maintenance burden, coordination overhead

These aren't arbitrary. They're the dimensions that experienced engineers have learned matter after shipping enough things to production.

## Getting Started

```bash
# Install
uv pip install greybeard

# Or with Anthropic support
uv pip install "greybeard[anthropic]"

# Configure
greybeard init

# See available packs
greybeard packs

# Review your current diff
git diff main | greybeard analyze
```

You'll need an API key from your LLM provider (OpenAI, Anthropic, or a local model via Ollama). `greybeard init` walks you through that.

The project is open source at [github.com/btotharye/greybeard](https://github.com/btotharye/greybeard) and on PyPI. Contributions are welcome — especially new content packs for domains I haven't covered yet. If you have domain expertise worth sharing, the pack format is intentionally simple: a few lines of YAML.
