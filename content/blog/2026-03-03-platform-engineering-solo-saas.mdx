## Who Platform Engineering Is Actually For

There's a common assumption that platform engineering — internal developer platforms, golden paths, developer experience programs — is for large engineering organizations. Teams big enough to justify dedicated platform teams.

I'd push back on that. The principles behind platform engineering matter at any scale. Self-service capabilities, catalog visibility, quality gates, infrastructure as code — these aren't luxuries for big teams. They're habits that keep small teams from drowning in operational overhead as the system grows.

This post walks through what I built for [Herp Ops](https://www.herpops.com), a reptile care SaaS I build alone, and why each piece is worth the setup time even at one engineer.

## The Stack

| Layer | Tool | What It Does |
|---|---|---|
| **Catalog** | [Port.io](https://getport.io) | Single source of truth — connects GitHub repos, deployments, code quality, and usage metrics into one queryable platform |
| **Infrastructure** | Terraform + Terraform Cloud | All catalog configuration as code, version-controlled, reviewed via PR — no click-ops |
| **Code Quality** | SonarCloud | Automated quality gates on every PR — catches real bugs before production |
| **CI/CD** | GitHub Actions + Railway + Vercel | Zero-config deployments on merge — fully automated from push to live |
| **Issue Tracking** | GitHub Projects | Lightweight backlog tied to code |
| **Backend** | Railway (FastAPI/Python) | Managed hosting with built-in observability |
| **Frontend** | Vercel (Next.js/TypeScript) | Edge-deployed frontend with preview deploys on every PR |

## The Software Catalog

Port.io is the hub. Everything connects to it — services, deployments, code quality, users, roadmap items.

The catalog has blueprints for:
- **Services** — `api`, `frontend`, `herp-ops-landing`, `herp-ops-infra`, `roadmap`, `herp-ops-ai`
- **Environments** — production and staging for each service
- **GitHub repos and PRs** — synced automatically via the GitHub Ocean integration
- **SonarCloud projects** — test coverage, bugs, vulnerabilities, code smells
- **Railway services** — deployment status, environment variables
- **Vercel deployments** — deployment status, domains
- **Users** — every user who signs up syncs to Port with subscription tier and usage stats
- **Beta signups** — the waitlist funnel

The AI agent in Port can answer questions across all of this:
> *"What's the code quality status for both services?"*
> *"How many users are on the Pro tier?"*
> *"Show me recent deployments"*

That's not a custom integration — it's GPT-4o with the catalog as context.

## Everything Is Code

Every blueprint, scorecard, entity, and action is defined in Terraform:

```hcl
# Example: Service blueprint definition
resource "port_blueprint" "service" {
  title       = "Service"
  icon        = "Service"
  description = "A microservice or deployed application"

  properties = {
    # GitHub integration syncs these
    repo_url   = "string"
    language   = "string"
    framework  = "string"
    # Quality gates
    test_coverage = "number"
    open_issues   = "number"
  }
}
```

The full structure looks like:

```text
terraform/port/
├── blueprints.tf    # all schema definitions
├── entities.tf      # seed data + import blocks
├── scorecards.tf    # quality rules
├── actions.tf       # self-service actions
├── main.tf          # provider config
└── variables.tf     # configurable values
```

Terraform Cloud runs `plan` and `apply` on every PR and merge. No click-ops in the Port UI — if you want to add a property to a blueprint, you open a PR.

This matters even as a solo builder. Without IaC, the catalog configuration drifts. You forget what you clicked, you can't review changes, and you can't reproduce the setup if you had to start over. With IaC, the git log is the audit trail.

<Callout type="info">
**Port v2 provider note:** Port's Terraform provider uses typed sub-maps (`string_props`, `number_props`, `boolean_props`) rather than a flat properties map. The migration from v1 is documented but takes an afternoon if you have existing blueprints.
</Callout>

## Scorecards: Automated Quality Gates

Port scorecards let you define rules that evaluate service quality and surface a Bronze/Silver/Gold status. My `production_readiness` scorecard checks:

**Bronze** — any service must have:
- CI/CD configured
- A README
- A health check endpoint

**Silver** adds:
- Test coverage ≥ 60%
- SonarCloud quality gate passing
- On-call runbook URL

**Gold** adds:
- Test coverage ≥ 80%
- No open critical Sonar issues
- Deployment documented

The API is at Gold (96.6% coverage). The frontend is at Silver (75.5% — working toward 80%). These aren't arbitrary targets — they're the questions I'd ask about any service I was about to hand off to someone else.

## Self-Service Actions

Port actions are forms that trigger GitHub Actions workflows. The ones I have:

- **scaffold_service** — creates a new repo from a template, wires CI/CD, registers in catalog
- **trigger_deployment** — redeploys a Railway service without leaving Port
- **sync_openapi_spec** — fetches the latest API spec and upserts it into Port
- **reset_demo_data** — resets the demo account in the app (useful for demos)
- **send_beta_invite** — marks a betaSignup entity as invited

At one engineer this is less about self-service (there's no one else to serve) and more about having automation that's discoverable, documented, and not living in a Notion doc nobody reads.

## Closing the Loop with Jira

The Jira integration (via Jira Ocean) closes a loop I find genuinely useful: users can file feature requests directly from the app's AI chat widget. The chat detects the intent, creates a Jira issue in the `HOFR` (Feature Requests) project, and the Jira Ocean integration syncs it into Port.

From Port I can see:
- What features users have asked for
- Which service the request relates to
- Whether it's been prioritized onto the roadmap

That's the kind of visibility that usually requires a product manager and a weekly sync meeting. With the right integrations, it just... exists.

> **Note:** After a few months with Jira, I'm planning to migrate back to GitHub Projects. Simpler for one person, and everything already lives in GitHub anyway.

## Was It Worth It?

Honest answer: some of it, immediately — some of it, not yet.

**Worth it immediately:**
- **SonarCloud quality gates** — caught real bugs before they hit production; paid for itself week one
- **Terraform for catalog config** — already saved me from drift twice; essential for reproducibility
- **Railway + Vercel + GitHub Actions** — zero-touch deploys are genuinely freeing; no manual deploy steps ever

**Worth it eventually (investment pays off as the system grows):**
- **Port catalog** — more useful as more services and integrations exist; will be essential before hiring
- **Scorecards** — enforce standards before technical debt compounds; critical if this becomes a team
- **Self-service actions** — handy now, invaluable once there are more people or more repetitive operations

**My recommendation:** Start with SonarCloud + GitHub Actions. Add Port + Terraform when you have 3+ services. Add Port scorecards when you're thinking about scaling the team. Most solo builders don't need Jira — GitHub Projects works fine.

The honest framing: this isn't about having a "real" platform engineering setup for clout. It's about not making decisions I'll regret in two years when the system is bigger and harder to change. The time to establish good habits is before you're in the middle of an incident wondering why nothing is observable.
