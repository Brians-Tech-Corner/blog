## The Difference Between a Chatbot and an Agent

Most AI coding tools answer questions. You ask, they respond. The conversation resets on the next session. There's no memory of what you decided last week, no awareness of what's deployed in production, no ability to actually *do* things — just answer things.

ProxiClaw is different. It's an AI agent that runs inside [OpenClaw](https://openclaw.ai) — a personal AI gateway — and it has persistent memory, tool access across my entire stack, and genuine context about what I'm working on. When I say "agent", I mean it in the literal sense: it takes action, not just provides information.

## What It Knows

ProxiClaw maintains memory across sessions. I don't re-explain context every time. It knows:

- What projects I'm working on and where the repos live
- What services are deployed where (Railway, Vercel, Terraform Cloud)
- My preferences (no useless filler words, direct answers, PRs for everything)
- Recent decisions and why I made them
- Open tasks and what's blocked

This comes from a set of workspace files it maintains itself: `SOUL.md` (who it is), `USER.md` (who I am), `MEMORY.md` (long-term memory), and daily notes it writes to track what happened.

## What It Can Do

The tool integrations are where it gets genuinely useful. ProxiClaw operates across three major domains:

**Code & Infrastructure:**
- Open a PR with a bug fix, including writing the tests
- Review a SonarCloud analysis and fix the flagged issues
- Resolve a merge conflict and push the result
- Update Terraform config and explain what changed

**Project & Ops:**
- File a Jira bug when I describe a problem
- Search Notion for notes I wrote two weeks ago
- Create a structured design doc from a bullet list
- Check Railway logs for errors and trigger deployments
- Run a database migration and watch the output

**Research & Discovery:**
- Pull documentation from a URL and summarize it
- Search for how other teams solve a specific problem
- Compare approaches and give a recommendation with tradeoffs

The key is that these aren't separate tools I have to switch between. I describe what I want in plain language and ProxiClaw figures out which tools to use and in what order.

## A Real Example

Here's a conversation that happened while building [Herp Ops](https://www.herpops.com), my reptile care SaaS:

> **Me:** The Port user sync is silently failing, I ran the bulk sync API call and it says it synced 4 users but only see 1 in Port
>
> **ProxiClaw:** *[checks the code, checks Port API directly, finds the issue]*
> The problem is in the admin router — the endpoint is registered as `/admin/sync-users-to-port` but the router prefix is already `/api/v1/admin`, so it's double-prefixing to `/api/v1/admin/admin/sync-users-to-port`. The sync was hitting the wrong path entirely. I've fixed the route path and removed a debug endpoint we don't want in prod — PR #81 is up.

That took about 30 seconds. The alternative was me digging through router code, checking Railway logs, and cross-referencing the Port API docs — maybe 20 minutes.

## The Memory System

Every session, ProxiClaw reads its memory files before doing anything. At the end of a session, it writes down what happened — decisions made, problems solved, things to remember next time.

Long-term memory (`MEMORY.md`) is curated — it distills the important stuff from the daily notes. Things like:

- The Port API uses POST for upserts, not PUT (learned the hard way)
- The SonarCloud org key is lowercase `herp-ops` not `Herp-Ops`
- Preferences I've stated once and shouldn't have to repeat

This sounds minor but it's the difference between an assistant that learns over time and one that starts from scratch every session.

## What It's Not

ProxiClaw isn't magic. It makes mistakes, especially on complex multi-step operations. It occasionally goes down a wrong path and needs a correction. The memory system helps but it's not perfect — sometimes context gets stale.

The right mental model is a **very capable junior engineer** who has full access to your systems, knows your preferences, and has been working alongside you for months. You still need to review their PRs. You still need to think through architecture. But the mechanical parts of engineering — the boilerplate, the debugging cycles, the cross-referencing documentation — those get dramatically faster.

## Getting Started

ProxiClaw runs inside [OpenClaw](https://openclaw.ai). The workspace files (`SOUL.md`, `USER.md`, `AGENTS.md`) set up its persona and context. Tool access is configured per-workspace — you add API tokens for the services you want it to have access to.

Setup takes about 30 minutes once you have your API tokens. The more context you give it upfront (what projects you're working on, what your preferences are, what tools you use), the better it gets from day one. Full setup docs are at [docs.openclaw.ai](https://docs.openclaw.ai).
